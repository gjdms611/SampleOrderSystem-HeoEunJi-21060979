# PRD 작성 (Explore 단계)

## 개요

Plan(`doc/plan/<기능명>.md`)/Action(구현)으로 넘어가기 전, 도메인 모델과 요구사항을 `doc/prd/<기능명>.md`에 문서화하라.

**핵심 원칙:** 요구사항이 불명확한 채로 PRD를 채우면, 그 잘못된 전제 위에서 Plan과 구현까지 통째로 다시 해야 한다. PRD 단계에서 사람 파트너와 두 번 확인하는 비용은, 구현까지 끝낸 뒤 되돌리는 비용보다 훨씬 싸다.

## 사용 시점

**항상 (Explore 단계 진입 시):**
- 이번 기능의 `doc/prd/<기능명>.md`가 아직 없음
- 도메인 규칙/상태머신/권한/메뉴/계산식 등 요구사항이 바뀌었는데 그 기능의 `doc/prd/<기능명>.md`가 갱신 안 됨

**예외 (사람 파트너에게 물어볼 것):**
- 사소한 오타/서식 수정처럼 도메인 규칙 변경이 없는 경우

## 절차

### 1. 요구사항 수집

원본 요구사항 문서(`요구사항.md` 등)와 사람 파트너와의 대화에서 아래 항목을 확인한다.

- 도메인 엔티티(속성표): 예) Sample(시료) 속성
- 상태머신: 예) Order(주문) 상태 전이와 각 상태의 의미
- 역할별 권한표
- 메인 메뉴/기능 목록
- 도메인 계산식/비즈니스 규칙

하나라도 불명확하면 추측하지 말고 사람 파트너에게 물어라. 애매한 부분을 "합리적으로 추정"해서 채우면, 그 추정이 틀렸을 때 PRD 전체가 틀린 전제 위에 서게 된다.

<Good>
"출고처리 메뉴에서 '부분 출고'가 가능한지 명시가 없습니다. 가능/불가능 여부 확인 부탁드립니다."
불명확한 지점을 콕 집어 질문
</Good>

<Bad>
"부분 출고는 일반적으로 지원하니 가능하다고 가정하고 작성하겠습니다."
확인 없이 흔한 패턴으로 추정 - 틀리면 PRD/Plan/구현이 전부 재작업
</Bad>

### 2. 요구사항 요약 확인 (인간 검토 #1)

전체 PRD.md를 다 쓰기 전에, 수집한 요구사항을 짧은 요약(bullet 목록 수준)으로 사람 파트너에게 먼저 보여주고 확인받는다. 이 시점에 방향이 틀렸으면 몇 줄만 고치면 되지만, PRD 전체를 다 쓴 뒤에 틀렸다는 걸 알면 문서 전체를 다시 써야 한다.

이 확인 없이 바로 PRD.md 전체를 작성하지 않는다.

### 3. PRD 작성/갱신 — `doc/prd/<기능명>.md`

PRD.md는 임시 산출물이 아니라 그 기능에 대한 요구사항 기록으로 계속 남아야 한다. 그래서 레포 루트가 아니라 `doc/prd/<기능명>.md`에 기능 단위로 작성한다. 최소 아래 섹션을 포함한다:

1. 목표/배경
2. 도메인 모델(엔티티 속성표)
3. 상태머신(다이어그램 또는 텍스트 표현 + 각 상태 설명)
4. 역할별 권한표
5. 메인 메뉴/기능 정의
6. 도메인 계산식/비즈니스 규칙
7. 미확정 사항(TBD) — 확인 필요한 항목을 숨기지 말고 명시

**파일 분리 규칙:** 지금 다루는 기능의 요구사항이 바뀌었다면 그 기능의 `doc/prd/<기능명>.md`만 갱신한다. 하지만 새로운 기능/사이클을 시작하는 것이라면 기존 `doc/prd/<기능명>.md`를 고쳐 쓰지 말고 그 기능 이름으로 된 새 파일을 만들어라. 과거 기능의 PRD 파일을 다른 기능 작업 중에 수정하면, 그 기능이 승인받았던 당시의 요구사항 기록이 사라져 나중에 "그때 왜 이렇게 정했는지" 되짚을 수 없게 된다.

### 4. 최종 검토 + 커밋 (인간 검토 #2)

작성된 PRD 문서 전체를 사람 파트너에게 제시하고 승인을 받는다. 승인 전에는 Plan 단계(`doc/plan/<기능명>.md` 작성, `tdd-skill` 트리거)로 넘어가지 않는다. 승인되면 `doc/prd/<기능명>.md`만 별도로 커밋한다.

## 완료 체크리스트

작업을 완료로 표시하기 전에:

- [ ] 도메인 엔티티/상태머신/권한/메뉴/계산식을 전부 확인했다 (추측으로 채운 항목 없음)
- [ ] 불명확한 항목은 TBD로 남기고 숨기지 않았다
- [ ] PRD 전체를 쓰기 전에 요약을 사람 파트너에게 보여주고 승인받았다 (인간 검토 #1)
- [ ] 완성된 PRD 문서를 사람 파트너에게 보여주고 승인받았다 (인간 검토 #2)
- [ ] `doc/prd/<기능명>.md`가 별도로 커밋되었다 (Plan/구현 커밋과 섞이지 않음)
- [ ] 다른 기능의 기존 `doc/prd/<기능명>.md`를 고쳐 쓰지 않았다 (이번 기능 파일만 새로 만들거나 갱신)

체크박스를 다 채울 수 없다면 아직 Explore 단계가 끝난 게 아니다. Plan 단계로 넘어가지 마라.

## 위험 신호 (Red Flags) - 멈추고 되돌아가라

- "일단 합리적으로 추정해서 PRD 채우고, 틀리면 나중에 고치죠" - 나중에 고치는 비용이 훨씬 크다
- "요약 확인은 생략하고 바로 PRD 전체를 쓸게요, 어차피 비슷할 거예요" - 그 "비슷함"이 틀렸을 때 전체 재작성
- "PRD는 초안이니 사람 검토 전에 Plan부터 같이 시작할게요" - Explore와 Plan이 뒤섞여 잘못된 전제가 Plan까지 전파됨
- "TBD로 남기지 않고 그냥 제일 흔한 케이스로 정해서 쓸게요" - 숨겨진 가정은 나중에 버그로 드러남
- "이번 기능은 저번 기능이랑 비슷하니까 저번 `doc/prd/<기능명>.md`를 고쳐서 재사용할게요" - 저번 기능의 승인된 요구사항 기록이 지워진다. 비슷해도 새 파일로 시작하라
